# 接口定义草案

这份文档的目标，是把“实时 Obsidian 同步 + snapshot”系统里的关键抽象边界提前固定下来。

这样做的价值不是形式化，而是避免你在实现早期就把同步逻辑、部署平台、数据库、对象存储、WebSocket 框架全耦合在一起。

如果这些接口先定义清楚，后续你可以：

- 替换部署平台
- 替换数据库
- 替换对象存储
- 替换 WebSocket 框架
- 但不用重写同步核心逻辑

## 1. 设计原则

先给几个总原则。

1. 接口表达能力边界，不表达供应商特性。
2. 同步核心只依赖抽象接口，不依赖 Node、Cloudflare、S3 SDK 等具体实现。
3. 接口要偏小，避免一个 `God service` 管一切。
4. 文本同步、blob、snapshot、缓存、传输必须分层。
5. 先保证正确性边界，再考虑性能优化。

## 2. 核心领域对象

这一节先统一一些术语。

### 2.1 VaultId

```ts
type VaultId = string;
```

表示一个同步 vault 的逻辑 ID。

### 2.2 FileId

```ts
type FileId = string;
```

表示 Markdown 文件的稳定身份。

### 2.3 BlobHash

```ts
type BlobHash = string;
```

通常是附件内容的 `sha256`。

### 2.4 BlobRef

```ts
interface BlobRef {
  hash: BlobHash;
  size: number;
  contentType?: string;
  updatedAt: string;
}
```

表示某个路径当前引用的 blob。

### 2.5 SnapshotMeta

```ts
interface SnapshotMeta {
  snapshotId: string;
  vaultId: VaultId;
  createdAt: string;
  schemaVersion?: number;
  markdownFileCount: number;
  blobFileCount: number;
  referencedBlobHashes?: BlobHash[];
}
```

### 2.6 PersistedDelta

```ts
interface PersistedDelta {
  seq: number;
  payload: Uint8Array;
  sha256: string;
  createdAt: string;
}
```

### 2.7 PersistedCheckpoint

```ts
interface PersistedCheckpoint {
  version: number;
  payload: Uint8Array;
  stateVector: Uint8Array;
  sha256: string;
  createdAt: string;
}
```

## 3. Transport 接口

`Transport` 负责“实时消息如何收发”，但不负责：

- 文档持久化
- blob 存储
- snapshot
- 合并语义

建议定义为两层：服务端会话接口 + 客户端连接接口。

### 3.1 服务端侧连接抽象

```ts
interface TransportSession {
  id: string;
  vaultId: VaultId;
  send(message: TransportMessage): Promise<void>;
  close(code?: number, reason?: string): Promise<void>;
  onMessage(handler: (message: TransportMessage) => Promise<void> | void): void;
  onClose(handler: () => Promise<void> | void): void;
}
```

### 3.2 客户端侧连接抽象

```ts
interface TransportClient {
  connect(input: ConnectInput): Promise<void>;
  disconnect(): Promise<void>;
  send(message: TransportMessage): Promise<void>;
  onMessage(handler: (message: TransportMessage) => Promise<void> | void): void;
  onStatusChange(handler: (status: ConnectionStatus) => void): void;
}
```

### 3.3 辅助类型

```ts
type ConnectionStatus =
  | "idle"
  | "connecting"
  | "connected"
  | "reconnecting"
  | "closed"
  | "failed";

interface ConnectInput {
  vaultId: VaultId;
  token: string;
  schemaVersion: number;
  deviceId: string;    // 设备的稳定唯一标识（UUID），用于 awareness、冲突溯源、离线状态管理
  deviceName?: string; // 可读的显示名，仅供 UI 展示
}
```

### 3.4 TransportMessage 建议

```ts
type TransportMessage =
  | HelloMessage           // 客户端 -> 服务端，握手发起 { type: "hello", deviceId: string, deviceName?: string, schemaVersion: number }
  | AuthOkMessage          // 服务端 -> 客户端，握手成功 { type: "auth_ok", serverSchemaVersion: number }
  | AuthFailedMessage      // 服务端 -> 客户端，鉴权失败 { type: "auth_failed", reason: string }
  | SchemaMismatchMessage  // 服务端 -> 客户端，schema 版本不兼容 { type: "schema_mismatch", serverSchemaVersion: number }
  | SyncUpdateMessage      // 双向，CRDT 增量 { type: "sync_update", update: Uint8Array }
  | SyncStateVectorMessage // 双向，用于请求缺失更新 { type: "sync_state_vector", sv: Uint8Array }
  | AwarenessMessage       // 双向，光标/在线状态 { type: "awareness_update", payload: Uint8Array }
  | RoomMetaMessage        // 服务端 -> 客户端，诊断信息 { type: "room_meta", meta: RoomMeta }
  | ErrorMessage;          // 服务端 -> 客户端，运行时错误 { type: "server_error", code: string, message: string }
```

关键点：

- 文本 update 应直接承载 `Uint8Array` 或可编码二进制
- 控制消息与 update 消息要分开
- schema mismatch 应在握手早期暴露，不应等到数据交换后才发现

## 4. RoomCoordinator 接口

`RoomCoordinator` 是每个 vault 的逻辑会合点。

它负责：

- 管理一个 vault 的在线会话
- 管理共享 `Y.Doc`
- 接收和广播更新
- 控制保存节奏

它不负责：

- 直接决定 SQLite 或 Postgres 怎么写
- 直接决定 S3 或 MinIO 怎么存 blob

### 4.1 接口定义

```ts
interface RoomCoordinator {
  readonly vaultId: VaultId;
  load(): Promise<void>;
  attachSession(session: TransportSession): Promise<void>;
  detachSession(sessionId: string): Promise<void>;
  applyClientUpdate(sessionId: string, update: Uint8Array): Promise<void>;
  applyAwareness(sessionId: string, payload: Uint8Array): Promise<void>;
  getMeta(): Promise<RoomMeta>;
  snapshotNow(triggeredBy?: string): Promise<SnapshotMeta>;
  disposeIfIdle(): Promise<boolean>;
}
```

### 4.2 RoomMeta

```ts
interface RoomMeta {
  vaultId: VaultId;
  schemaVersion?: number;
  connectedClientCount: number;
  loaded: boolean;
  updatedAt?: string;
}
```

### 4.3 RoomManager

```ts
interface RoomManager {
  getOrCreate(vaultId: VaultId): Promise<RoomCoordinator>;
  get(vaultId: VaultId): Promise<RoomCoordinator | null>;
  disposeIdleRooms(): Promise<void>;
}
```

## 5. DocumentStore 接口

`DocumentStore` 是最关键的后端抽象之一。

它负责：

- 持久化共享文档的 checkpoint
- 持久化 journal 增量
- 从 durable store 重建文档

它不负责：

- 管理 WebSocket 会话
- 管理 blob
- 处理磁盘桥接

### 5.1 接口定义

```ts
interface DocumentStore {
  load(vaultId: VaultId): Promise<LoadedDocumentState>;
  appendDelta(input: AppendDeltaInput): Promise<AppendDeltaResult>;
  writeCheckpoint(input: WriteCheckpointInput): Promise<void>;
  readMeta(vaultId: VaultId): Promise<DocumentStoreMeta | null>;
  writeMeta(meta: DocumentStoreMeta): Promise<void>;
  // 由 RoomCoordinator 在每次 saveNow() 末尾调用，判断 journal 是否达到压缩阈值，
  // 若达到则生成新 checkpoint 并清理旧 journal 段。调用方需在压缩期间暂停新增 delta 写入。
  compactIfNeeded(vaultId: VaultId): Promise<CompactResult>;
}
```

### 5.2 输入输出类型

```ts
interface LoadedDocumentState {
  checkpoint: PersistedCheckpoint | null;
  deltas: PersistedDelta[];
  meta: DocumentStoreMeta | null;
}

interface AppendDeltaInput {
  vaultId: VaultId;
  payload: Uint8Array;
  expectedNextSeq?: number;
}

interface AppendDeltaResult {
  seq: number;
  entryCount: number;
  totalBytes: number;
}

interface WriteCheckpointInput {
  vaultId: VaultId;
  checkpoint: PersistedCheckpoint;
  replaceUpToSeq?: number;
}

interface DocumentStoreMeta {
  vaultId: VaultId;
  currentCheckpointVersion: number;
  nextSeq: number;
  journalEntryCount: number;
  journalTotalBytes: number;
  updatedAt: string;
}

interface CompactResult {
  compacted: boolean;
  checkpointVersion?: number;
}
```

### 5.3 正确性要求

- `appendDelta` 必须保证顺序性
- `load` 返回的 deltas 必须是严格有序的
- checkpoint 和 state vector 必须一起持久化
- 任意损坏不能静默跳过

## 6. BlobStore 接口

`BlobStore` 负责附件和对象归档。

它负责：

- 检查 blob 是否存在
- 上传 blob
- 下载 blob
- 可选列举对象

它不负责：

- 路径绑定关系
- 文本 merge
- snapshot 业务流程

### 6.1 接口定义

```ts
interface BlobStore {
  has(vaultId: VaultId, hash: BlobHash): Promise<boolean>;
  // 返回入参中"已存在于存储层"的 hash 子集，用于批量上传前的去重检查
  hasMany(vaultId: VaultId, hashes: BlobHash[]): Promise<Set<BlobHash>>;
  put(input: PutBlobInput): Promise<void>;
  get(input: GetBlobInput): Promise<StoredBlob | null>;
  delete?(vaultId: VaultId, hash: BlobHash): Promise<void>;
}
```

### 6.2 输入输出类型

```ts
interface PutBlobInput {
  vaultId: VaultId;
  hash: BlobHash;
  bytes: Uint8Array;
  contentType?: string;
}

interface GetBlobInput {
  vaultId: VaultId;
  hash: BlobHash;
}

interface StoredBlob {
  hash: BlobHash;
  bytes: Uint8Array;
  contentType?: string;
  size: number;
}
```

### 6.3 设计建议

- blob 推荐内容寻址
- blob 本身推荐不可变
- 删除应谨慎，最好延后或经 GC 确认后再做

## 7. SnapshotStore 接口

`SnapshotStore` 负责存放 snapshot archive 和 index。

它负责：

- 写入 snapshot 归档
- 列出 snapshot
- 获取 snapshot 内容

它不负责：

- 决定 snapshot 何时创建
- 决定恢复策略

### 7.1 接口定义

```ts
interface SnapshotStore {
  put(input: PutSnapshotInput): Promise<SnapshotMeta>;
  list(vaultId: VaultId): Promise<SnapshotMeta[]>;
  get(vaultId: VaultId, snapshotId: string): Promise<StoredSnapshot | null>;
}
```

### 7.2 输入输出类型

```ts
interface PutSnapshotInput {
  vaultId: VaultId;
  snapshotId: string;
  payload: Uint8Array;
  meta: SnapshotMeta;
}

interface StoredSnapshot {
  meta: SnapshotMeta;
  payload: Uint8Array;
}
```

### 7.3 建议

- payload 建议存压缩后的全量共享状态
- meta 与 payload 分开存储更便于列表查询

## 8. LocalCache 接口

`LocalCache` 是客户端抽象。

它负责：

- 缓存最近一次共享状态
- 支持离线后恢复
- 保存部分本地同步辅助状态

它不负责：

- 做全局权威持久化
- 替代服务端 DocumentStore

### 8.1 接口定义

```ts
interface LocalCache {
  load(vaultId: VaultId): Promise<LocalCacheState | null>;
  save(vaultId: VaultId, state: LocalCacheState): Promise<void>;
  clear(vaultId: VaultId): Promise<void>;
}
```

### 8.2 类型定义

```ts
interface LocalCacheState {
  vaultId: VaultId;
  ydocUpdate: Uint8Array;
  updatedAt: string;
}
```

## 9. FilesystemBridge 接口

这是插件侧非常关键的一层。

它负责：

- 将磁盘变更导入共享状态
- 将共享状态写回磁盘
- 做自写抑制和路径串行化

### 9.1 接口定义

```ts
interface FilesystemBridge {
  // 磁盘 -> 共享状态
  markDirty(path: string): void;                                              // 将路径加入 dirty set，等待异步处理
  drain(): Promise<void>;                                                     // 消费 dirty set，读取最新磁盘内容，将差异导入共享状态

  // 共享状态 -> 磁盘
  flushFile(path: string): Promise<void>;                                     // 将指定路径的共享状态写回本地磁盘（串行，防止并发覆盖）
  suppressExpectedWrite(path: string, fingerprint: FileFingerprint): void;    // 注册预期的自写内容指纹，后续文件事件若匹配则视为回声并抑制

  // 路由
  resolveVaultId(path: string): VaultId;  // 根据文件路径判断归属哪个 room（主 vault 或某个共享目录挂载）
}
```

### 9.2 辅助类型

```ts
interface FileFingerprint {
  sha256: string;
  byteLength: number;
}
```

### 9.3 约束

- 不允许按时间窗口盲抑制
- 必须以内容指纹确认自写回声
- 同一路径必须串行写回

## 10. SyncEngine 接口

`SyncEngine` 是插件内部的编排层。

它依赖上面这些接口，把系统真正串起来。

### 10.1 接口定义

```ts
interface SyncEngine {
  start(): Promise<void>;
  stop(): Promise<void>;
  // 全量协调：扫描本地文件与共享状态的差异，补齐双向缺失。
  // 通常在启动、重连、或用户手动触发时调用，不用于平时的增量同步。
  reconcile(): Promise<void>;
  handleLocalFileChange(path: string): Promise<void>;
  handleRemoteUpdate(update: Uint8Array): Promise<void>;
  createSnapshot(): Promise<SnapshotMeta>;
  restoreSnapshot(snapshotId: string): Promise<void>;
}
```

### 10.2 它依赖什么

- `TransportClient`
- `LocalCache`
- `FilesystemBridge`
- `BlobStore` 的客户端适配器
- `SnapshotStore` 的客户端适配器

### 10.3 它不应该直接依赖什么

- SQLite 驱动
- S3 SDK
- 特定 WebSocket 库实例

这些应该被封装在适配层后面。

## 11. 共享目录接口

共享目录是"给某个本地子目录分配独立 VaultId + 独立鉴权"的能力，实现不同 vault 之间共享部分内容。它不引入新的同步机制，而是复用现有的 room 模型。

### 11.1 配置类型

```ts
interface SharedDirectoryMount {
  localPath: string;     // 相对 vault 根的挂载路径，e.g. "Shared"
  vaultId: VaultId;     // 该共享目录在服务端对应的逻辑 room ID
  token: string;        // 访问该共享目录 room 的鉴权凭据（独立于主 vault token）
  serverUrl?: string;   // 可选，若共享目录托管在不同服务器
  readOnly?: boolean;   // 可选，只读挂载（本地修改不上传）
}

// 插件的完整同步配置
interface SyncConfig {
  primaryVaultId: VaultId;
  primaryToken: string;
  serverUrl: string;
  deviceId: string;
  sharedMounts?: SharedDirectoryMount[];
}
```

### 11.2 路径语义约定

共享目录内的文件路径相对于**共享目录根**，而不是各自 vault 的根。

例如：vault A 把共享目录挂在 `Shared/`，vault B 挂在 `团队/`。共享 room 里的文件路径是 `notes/foo.md`，在 vault A 本地表现为 `Shared/notes/foo.md`，在 vault B 本地表现为 `团队/notes/foo.md`。这样文件身份在不同 vault 间保持一致。

FilesystemBridge 在处理共享目录的文件时，需要剥离本地挂载前缀后再与共享模型交互。

### 11.3 跨目录移动的处理

如果用户把文件从主 vault 目录移动到共享目录（或反向），FilesystemBridge 应将其识别为两个独立操作：

1. 在原 room（主 vault）中删除该文件
2. 在目标 room（共享目录）中创建该文件

这不是同一 room 内的 rename，而是跨 room 的迁移。实现上可通过检测事件的源路径和目标路径是否跨越挂载边界来判断。

### 11.4 隔离要求

- 主 vault 的 `Y.Doc` 和共享目录的 `Y.Doc` 必须完全独立，不得共用同一个文档对象
- 各自的 snapshot、journal、LocalCache 条目按 vaultId 隔离
- 服务端鉴权：共享目录的 token 只对其对应的 vaultId 有效，不能访问主 vault

## 12. 推荐的依赖方向

建议依赖方向如下：

```text
UI / Plugin Lifecycle
        -> SyncManager              # 编排主 vault + 所有共享目录挂载
        -> SyncEngine (per vault)
        -> TransportClient
        -> FilesystemBridge
        -> LocalCache

Server Entry
        -> RoomManager
        -> RoomCoordinator
        -> DocumentStore
        -> BlobStore
        -> SnapshotStore
```

关键规则：

- `RoomCoordinator` 可以依赖 `DocumentStore` / `SnapshotStore`
- `DocumentStore` 不能反向依赖 `RoomCoordinator`
- `SyncEngine` 可以调用 transport，但 transport 不知道同步业务细节
- `SyncManager` 负责按路径前缀将文件事件路由到正确的 `SyncEngine` 实例

## 13. 一组最小实现映射

如果你现在就要开始实现，这些接口可以先映射成：

- `TransportClient` -> 浏览器/WebSocket 客户端
- `TransportSession` -> Node `ws` 的连接包装器
- `RoomManager` -> 内存 `Map<VaultId, VaultRoom>`
- `DocumentStore` -> `SQLiteDocumentStore`
- `BlobStore` -> `S3BlobStore` 或 `MinioBlobStore`
- `SnapshotStore` -> `S3SnapshotStore`
- `LocalCache` -> `IndexedDbLocalCache`
- `FilesystemBridge` -> `ObsidianFilesystemBridge`
- `SyncManager` -> 持有主 vault `SyncEngine` + `SharedDirectoryMount[]` 对应实例的编排器

这已经足够搭出第一版。

## 14. 最后建议

如果你准备真的写代码，推荐顺序是：

1. 先把接口和类型定义独立出来
2. 再写最简实现
3. 再接入真实 SQLite / S3 / WebSocket
4. 最后再做性能优化和增强功能

不要反过来先写一堆具体实现，再回头抽象接口。

那样通常会把同步核心、存储细节、平台细节粘死在一起，后面很难拆。
