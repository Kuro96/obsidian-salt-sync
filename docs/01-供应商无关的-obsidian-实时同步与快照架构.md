# 供应商无关的 Obsidian 实时同步与快照架构

这份文档不讨论 Cloudflare、AWS、阿里云或任何特定厂商。

目标只有两个：

1. 说明构建“实时 Obsidian 同步 + snapshot”到底需要哪些核心技术能力。
2. 说明这些能力应该如何编排，才能形成一个可用、可恢复、可扩展的系统。

核心原则是：

- 同步系统依赖的是能力，不是供应商。
- 供应商产品只是这些能力的一种承载方式。
- 只要能力边界设计得对，部署目标可以替换。

## 1. 先定义问题

一个真正可用的 Obsidian 同步系统，要同时处理四个世界：

1. 本地文件系统
2. 编辑器里的实时文本状态
3. 多设备之间共享的复制状态
4. 可持久化、可恢复的后端存储

这四层的语义完全不同：

- 文件系统事件是嘈杂的、延迟的、可能乱序的
- 编辑器操作是高频、细粒度、强交互的
- 多端同步需要可合并、可收敛的复制模型
- 持久化要求可恢复、可校验、不能写放大失控

如果把它们粗暴地缝在一起，系统就会出现：

- 自写回声
- 冲突副本
- 远端覆盖本地
- 外部编辑丢失
- 重启后状态不一致
- 无法做可靠恢复

所以，正确做法不是“选一个服务”，而是先拆解需要的能力。

## 2. 核心技术栈分别提供什么能力

下面按能力层来讲。

### 2.1 宿主集成层

典型技术：

- Obsidian Plugin API

它提供的能力：

- 读取和写入 vault 内文件
- 监听 create / modify / delete / rename 等文件事件
- 获取当前打开的编辑器实例
- 注册命令、设置页、状态提示
- 与 Obsidian 的生命周期对接

为什么必须有：

- 你的同步系统不是在抽象文档库里运行，而是在 Obsidian 里运行。
- 如果拿不到文件事件和编辑器状态，就没法做“原生”的同步体验。

它不应该负责的事情：

- 不应该定义分布式合并语义
- 不应该直接承担最终真相来源

### 2.2 文本复制与合并层

典型技术：

- Yjs
- Automerge
- 其他支持文本操作的 CRDT

它提供的能力：

- 多端并发编辑下的自动合并
- 离线后重连的收敛能力
- 因果有序的更新表达
- 与具体传输层解耦的 update 模型

为什么必须有：

- 传统文件同步擅长“搬文件”，不擅长“实时共同编辑同一份文本”。
- 如果没有 CRDT 或等价机制，系统最终只能退化到“最后写入者覆盖”或“冲突副本”。

它不应该负责的事情：

- 不应该承载大体积二进制附件
- 不应该直接处理对象存储、磁盘抑制、文件事件降噪

### 2.3 编辑器绑定层

典型技术：

- CodeMirror 6 与 Yjs 的 binding

它提供的能力：

- 把当前打开的编辑器和共享文本状态直接绑定
- 支持低延迟实时输入传播
- 支持更细粒度的编辑历史保留
- 支持光标、选区、意识状态扩展

为什么必须有：

- 如果只靠磁盘事件同步，实时体验会很差。
- 编辑器绑定是“真正在输入时就同步”的关键。

它不应该负责的事情：

- 不覆盖整个 vault，只覆盖当前打开文档
- 不替代文件系统桥接层

### 2.4 文件系统桥接层

典型技术与策略：

- dirty set / 脏路径队列
- 按路径串行写回
- 内容 hash 指纹确认
- 基于 diff 的导入策略

它提供的能力：

- 把磁盘上的外部编辑安全导入共享状态
- 把远端同步结果安全写回本地文件
- 消除自写回声
- 抵御文件系统事件风暴和乱序问题

为什么必须有：

- Obsidian vault 本质上仍然是普通文件夹。
- 用户可能用 git、脚本、AI agent、外部编辑器直接修改文件。
- 如果没有这一层，系统只对“插件内部编辑”有效，对真实世界无效。

它不应该负责的事情：

- 不应用时间窗口当正确性依据
- 不应该看到远端变化就直接粗暴覆盖磁盘

### 2.5 本地持久缓存层

典型技术：

- IndexedDB
- SQLite
- 其他嵌入式 KV 或本地数据库

它提供的能力：

- 离线编辑期间的本地状态保留
- 启动时的快速恢复
- 重连后的 merge 连续性

为什么必须有：

- 真实同步系统不能假设网络永远在线。
- 如果每次启动都完全依赖远端重建，体验和可靠性都会差很多。

它不应该负责的事情：

- 不应被视为全局真相来源
- 它是本地 durability 和启动优化层

### 2.6 实时传输层

典型技术：

- WebSocket
- WebRTC 加 relay
- 任意双向低延迟消息通道

它提供的能力：

- 长连接实时传输更新
- awareness、presence、广播等低延迟消息流
- 比轮询和整文件上传更低的同步延迟

为什么必须有：

- 实时协作本质上是流式复制问题，不是定时上传问题。

它不应该负责的事情：

- 不定义持久化
- 不定义合并规则

### 2.7 room 协调层

典型实现：

- 每个 vault 一个 actor
- 每个 vault 一个单线程 room 进程
- 基于 vaultId 的 sticky 路由 room 服务

它提供的能力：

- 为每个 vault 提供一个逻辑上的会合点
- 聚合来自多客户端的更新
- 统一向订阅者广播更新
- 串行化 room 内的一些持久化和元数据操作

为什么必须有：

- 当多个设备同时在线时，需要一个清晰的“room 级协调边界”。
- 这是组织加载、保存、广播、诊断的天然位置。

它不应该负责的事情：

- 不应该演化成一个全局超级单例
- 不应该把数据模型耦合死在某个云产品上

### 2.8 文档持久化层

典型设计：

- checkpoint + journal
- 快照 + 增量日志
- append-mostly 的存储引擎

它提供的能力：

- 进程重启后的恢复
- 崩溃后的状态回放
- 降低每次保存都全量重写的写放大
- 对复制状态做完整性校验

为什么必须有：

- room 不是永生的。
- 网络服务会重启、迁移、扩容、崩溃。
- 没有 durable persistence，实时同步只是“内存聊天室”。

它不应该负责的事情：

- 不应该假设内存状态始终可靠
- 不应该每次保存都重写整个 CRDT 图

### 2.9 对象存储层

典型技术：

- S3
- MinIO
- Backblaze B2
- 任何兼容对象存储

它提供的能力：

- 存放附件二进制数据
- 存放 snapshot 归档
- 通过内容寻址做去重
- 低成本按 key 获取 blob

为什么必须有：

- 二进制附件和文本 CRDT 的成本模型完全不同。
- 把 PDF、图片、视频塞进文本同步模型是错误设计。

它不应该负责的事情：

- 不负责实时文本合并
- 不负责笔记级并发冲突语义

### 2.10 snapshot 子系统

典型形式：

- 压缩后的全量共享状态归档
- 配套 metadata index
- 可选的附件引用清单

它提供的能力：

- 时间点恢复
- 用户可感知的备份与回滚
- 对逻辑错误、批量误操作、坏导入的兜底

为什么必须有：

- “能收敛”不等于“能恢复”。
- 如果错误状态已经同步到所有端，系统依然需要救援通道。

它不应该负责的事情：

- 不替代实时同步主链路
- 不应该变成平时编辑的主要数据路径

## 3. 这些技术栈应该如何编排

下面讲真正的系统编排。

### 3.1 本地层：vault 仍然是普通文件

用户依然拥有普通的本地 Markdown 文件和附件。

插件需要监听：

- 文件创建、修改、删除、重命名
- 当前打开的编辑器
- 插件设置和设备标识

这一层的职责是“接入用户真实工作流”，不是“定义最终合并”。

### 3.2 共享模型层：建立 vault 级复制状态

系统需要维护一个共享的 vault 模型，通常包含：

- 稳定文件 ID
- 路径映射
- 每个 Markdown 文件对应的文本对象
- 附件引用关系
- tombstone / 删除标记
- schemaVersion 等系统元数据

**稳定文件 ID 的工作方式：** 文件创建时分配一个 UUID 作为 `fileId`，同时在共享模型中维护双向映射 `pathToId` 和 `idToPath`。rename 操作只更新这两张映射表，`fileId` 本身不变，绑定在该 ID 上的文本内容和编辑历史因此完整保留。

这里有两种设计：

1. 单 vault 共享文档
2. 每文件一个子文档再加总索引

对 Obsidian 个人笔记场景，单 vault 共享文档通常更合适。理由如下：

- 重命名、移动文件夹这类跨文件操作更容易原子化——不需要跨多个子文档协调
- snapshot 只需导出一份共享状态，逻辑简单
- room 重启恢复时只加载一个文档，冷启动代价可预测

权衡点：vault 包含数千个文件时，单文档的内存占用会更高。如果遇到这个规模，可以考虑按目录分片，但第一版不需要提前设计这个。

### 3.3 编辑器路径：打开的 Markdown 走实时绑定

当用户打开某个 Markdown 文件时：

1. 找到该文件在共享模型中的文本对象
2. 将编辑器与该文本对象绑定
3. 用户输入直接写入共享文本状态
4. 远端更新直接反馈到编辑器

这条路径负责“实时性”。

### 3.4 磁盘路径：未打开文件与外部编辑走桥接

磁盘到共享状态：

1. 文件事件发生时，不立刻处理全部事件
2. 先把路径加入 dirty set
3. 由异步 drainer 按节奏读取最新内容
4. 如果内容指纹与预期自写内容一致，则抑制
5. 否则把差异导入共享文本状态

共享状态到磁盘：

1. 根据共享状态生成目标文件内容
2. 对同一路径做串行写回
3. 写回前记录预期内容指纹
4. 写入磁盘
5. 后续文件事件只有在内容精确匹配时才视为自写回声

这条路径负责“兼容真实文件世界”。

### 3.5 网络路径：每个 vault 对应一个 room

每个客户端按 `vaultId` 连接到一个 room。

room 负责：

- 加载当前共享状态
- 接收客户端 update
- 广播给其他客户端
- 控制 save 节奏
- 暴露 schema / 诊断元数据

这条路径负责”多设备会合与实时广播”。

**可扩展场景：共享目录。** 如果需要在不同 vault 之间同步某个目录，可以给该共享目录分配独立的 `vaultId`，并配置独立的鉴权凭据。客户端同时维护主 vault 的 room 连接和共享目录的 room 连接，两者完全独立。共享目录内的文件路径相对于共享目录根，而不是各自 vault 的根——这样不同 vault 把它挂载在不同本地路径时，文件身份仍然一致。

### 3.6 持久化路径：checkpoint + journal

共享状态的持久化不应该是“每次全量保存”，而应该是：

1. 周期性或条件触发地写 checkpoint
2. 平时把增量写进 journal
3. journal 达到阈值后再 compact 成新 checkpoint

恢复时：

1. 读取 checkpoint
2. 校验完整性
3. 顺序回放 journal
4. 还原出最新共享状态

这条路径负责“可恢复性和成本控制”。

### 3.7 附件路径：内容寻址，不走文本 CRDT

附件同步应该独立设计：

1. 客户端对附件内容做 hash
2. 检查对象存储中是否已存在该 blob
3. 如不存在则上传
4. 在共享模型中只记录附件引用元数据
5. 其他设备观察到该引用后再按需拉取
6. 拉取后镜像到本地磁盘

这样做的收益：

- 降低共享文本状态的内存压力
- 支持 dedup
- 降低 snapshot 和恢复复杂度

推荐原则：

- blob 是不可变对象
- 路径到 blob 的绑定是可变元数据

### 3.8 snapshot 路径：从共享模型生成恢复点

snapshot 不应来自“临时扫一遍文件夹”，而应该来自共享模型。

推荐结构：

- 压缩后的共享状态归档
- snapshot 元数据索引
- 可选的附件 hash 引用列表

创建 snapshot：

1. 从共享模型导出全量状态
2. 压缩
3. 写入对象存储
4. 写入 metadata index
5. 可选记录当时引用到的附件集合

恢复 snapshot：

1. 读取 snapshot 归档
2. 解码和检查元数据
3. 与当前共享状态做 diff
4. 允许用户全量恢复或选择性恢复
5. 通过共享模型和文件系统桥接把恢复结果重新 materialize 到本地

这条路径负责“灾难恢复和历史回退”。

## 4. 关键不变量

如果想让系统在真实使用中保持正确，下面这些规则最好不要破：

1. Markdown 文本通过 CRDT 同步，而不是时间戳覆盖。
2. 二进制附件不进入文本 CRDT。
3. 磁盘与共享状态之间用内容确认抑制，而不是时间窗口抑制。
4. 同一路径只能有一条串行磁盘写链。
5. 同一时刻每个 vault 只有一个逻辑 room 协调边界。
6. 持久化应是 append-mostly，再周期 compact，而不是每次全量重写。
7. snapshot 应从共享模型生成，而不是事后用边缘信息拼出来。
8. 本地缓存有用，但不应被误当作全局权威。
9. 恢复路径应走同一套状态机，而不是绕过同步逻辑直接乱写文件。

## 5. 最小可移植蓝图

如果完全不想和供应商耦合，一个最小但合理的技术组合是：

- Obsidian Plugin API
- Yjs 这样的文本 CRDT
- CodeMirror 绑定
- IndexedDB 或 SQLite 本地缓存
- WebSocket 实时传输
- 每个 vault 一个 room 协调器
- SQLite 或 Postgres 存 checkpoint + journal
- S3 兼容对象存储保存附件和 snapshot

这套组合已经足够实现：

- 实时 Markdown 同步
- 离线合并和重连
- 外部编辑兼容
- 附件同步
- snapshot 创建与恢复

## 6. 为了可替换部署，代码边界应该怎样设计

如果你不想被某个平台绑定，代码里最好显式抽出这些接口：

- `Transport`
- `RoomCoordinator`
- `DocumentStore`
- `BlobStore`
- `SnapshotStore`
- `LocalCache`

让“同步逻辑”依赖这些接口，而不是依赖具体云平台 runtime。

这样做的结果是：

- 你可以换部署平台
- 你可以换对象存储
- 你可以换持久化后端
- 但不会被迫重写整个同步模型

这才是真正的供应商无关。
